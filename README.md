## 📋 주요기능 상세 설명
루팡샵의 기능 요구 명세서를 기준으로, 각 메뉴별 주요 기능을 아래에 정리하였습니다.
## 📋 주요기능 상세 설명

### **🛍️ 구매자 기능 (ConsumerMenu)**

**• 아이템 목록**  
→ 등록된 모든 아이템을 확인합니다.  

**• 아이템 구매하기**  
→ 선택한 아이템을 바로 구매합니다.  

**• 판매자 닉네임으로 검색**  
→ 판매자의 이름으로 아이템을 찾습니다.  

**• 장바구니**  
→ 장바구니 메뉴로 이동합니다.  

**• 아이템 검색**  
→ 키워드로 아이템을 검색합니다.  

**• 판매하기**  
→ 판매자 메뉴로 진입합니다.  

**• 수령 확인**  
→ 구매한 아이템의 수령을 완료 처리합니다.  

**• 구매 취소**  
→ 거래 중인 아이템을 취소합니다.  

**• 내 정보 보기**  
→ 마이페이지로 이동하여 회원 정보를 관리합니다.  

---

### **🧾 판매자 기능 (SellerMenu)**

**• 내 등록 아이템 보기**  
→ 판매자가 등록한 아이템 목록을 조회합니다.  

**• 아이템 등록**  
→ 판매자가 아이템을 직접 등록할 수 있습니다.  

**• 아이템 수정**  
→ 등록한 아이템의 정보를 수정합니다.  

**• 판매 중지**  
→ 더 이상 판매하지 않을 아이템을 중지시킵니다.  

**• 예약/완료 상태 아이템 보기**  
→ 예약 중이거나 거래 완료된 아이템을 확인합니다.  

---

### **🛒 장바구니 기능 (CartMenu)**

**• 장바구니에 담기**  
→ 원하는 아이템을 장바구니에 추가합니다.  

**• 장바구니 목록 보기**  
→ 현재 보관 중인 아이템을 확인합니다.  

**• 아이템 제거**  
→ 선택한 아이템을 장바구니에서 삭제합니다.  

**• 장바구니 비우기**  
→ 장바구니 내 모든 아이템을 한 번에 삭제합니다.  

**• 결제하기 (전체 / 선택)**  
→ 장바구니의 전체 또는 일부 아이템을 결제합니다.  

---

### **👤 마이페이지 기능 (MyPageMenu)**

**• 닉네임 변경**  
→ 사용자 닉네임을 수정합니다.  

**• 비밀번호 변경**  
→ 계정 로그인 비밀번호를 변경합니다.  

**• 회원 탈퇴**  
→ 회원이 직접 탈퇴를 진행할 수 있습니다.  

**• 포인트 충전**  
→ 결제용 포인트를 충전합니다.  

---

### **🛠️ 관리자 기능 (AdminMenu)**

**• 전체 회원 목록 조회**  
→ 모든 가입 회원의 정보를 확인합니다.  

**• 전체 아이템 목록 조회**  
→ 전체 등록 아이템을 확인하고 관리합니다.  

**• 전체 거래 목록 조회**  
→ 진행 중이거나 완료된 거래 내역을 확인합니다.  

**• 거래 상태 수동 변경**  
→ 거래 상태를 관리자가 직접 변경할 수 있습니다.  

**• 회원 강제 탈퇴**  
→ 운영 정책에 따라 회원을 강제로 탈퇴시킬 수 있습니다.  

**• 아이템 삭제**  
→ 관리자가 특정 아이템을 삭제합니다.  

**• 게임별 아이템 수량 확인**  
→ 게임 이름을 기준으로 아이템 재고량을 통계로 확인합니다.

## 🧩 직접 개발하며 마주한 고민과 배움

**🔗 메뉴 흐름 및 역할 기반 기능 설계 :**
게임 아이템 거래 플랫폼인 루팡샵은 사용자, 판매자, 관리자 세 가지 역할이 명확히 나뉘는 구조입니다.
각 역할에 적절한 기능을 어디에 배치하고 어떻게 연결할지를 정하는 과정이 가장 까다로웠습니다.
로그인 분기, 권한 조건, 메뉴 간 이동 등을 조정하느라 전체 메뉴 구조를 여러 차례 다시 설계했습니다.


**🔍 아이템 검색 및 필터링 조건 구현 :**
게임명, 가격, 등록자 등 다양한 조건으로 아이템을 검색할 수 있도록 구현해야 했습니다.
이 조건들을 조합하여 필터링하고, 결과를 콘솔 UI에 정돈되게 출력하는 과정이 생각보다 복잡했습니다.
검색 조건 클래스와 출력 로직 간의 정합성을 맞추는 데에도 꽤 많은 시행착오가 있었습니다.


**⚠️ 입력 예외 처리 및 피드백 메시지 설계 :**
잘못된 입력 상황에서도 프로그램이 종료되지 않고
자연스럽게 안내 메시지를 통해 다시 입력을 유도할 수 있도록 설계해야 했습니다.
단순한 try-catch 수준을 넘어서 UX 흐름 전반을 고려한 구조 설계가 필요했습니다.


**😵‍💫 기능 테스트 반복 :**
기능이 많아질수록 하나의 기능 수정이 다른 기능에 영향을 주는 경우가 많았습니다.
특히 거래 상태 변경, 장바구니, 포인트 결제 기능은 연결된 부분이 많아
수작업 시나리오 테스트를 여러 번 반복해야 했습니다.

## ⚙️ 다음에 개선하고 싶은 부분

**🔁 중복 조건 분기의 함수화**  
사용자 권한, 아이템 상태 등 반복되는 조건문이 많았고, 이를 공통 메서드로 분리해 로직의 일관성과 유지보수성을 높일 필요가 있다고 느꼈습니다.

**🧪 기능 증가에 따른 복잡도 증가**  
기능이 늘어날수록 코드의 길이와 조건 분기가 복잡해졌고, 특정 기능 수정 시 다른 부분에 영향을 주는 경우가 많았습니다. 기능 단위 분리와 테스트 체계 도입이 절실히 필요합니다.

**🖥️ UI/UX 출력 통일성 부족**  
출력 형식, 공백, 안내 메시지가 메뉴마다 제각각이라 사용자 경험에 혼선을 줬습니다. 출력 포맷과 메시지 규칙을 통일하는 개선이 필요합니다.

## 🧾 회고 총평
이번경험을 통해 단순한 기능 구현을 넘어, 역할 기반 기능 분리와 계층 구조 설계에 대한 실전 감각을 익힐 수 있었습니다.
사용자, 판매자, 관리자 각각의 역할에 맞는 기능을 구조적으로 나누고, VO–DAO–Service 간 데이터 흐름을 직접 설계하면서 책임 분리와 유지보수의 중요성을 체감했습니다.

또한 콘솔 환경이라고 해도 UX 설계와 출력 포맷의 일관성이 사용자 경험에 중요한 영향을 미친다는 점을 배웠습니다.
입력 예외 처리, 안내 메시지 흐름, 출력 정렬 등 사용자가 헤매지 않도록 유도하는 흐름 설계가 실제 사용성과 직결됨을 느꼈습니다.

기능이 많아질수록 테스트 체계의 필요성도 절실해졌습니다.
